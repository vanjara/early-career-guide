# Interview Prep — Tech Track
## SWE, Data Engineering, Data Science

## The process

LeetCode-style coding problems, 1-2 per round, 45 minutes, you talk through your thinking out loud.

What interviewers are evaluating:
- Can you break down a problem?
- Do you communicate your approach before coding?
- Can you identify edge cases?
- Do you write clean, working code?

---

## What to focus on

```
Must know          Arrays, strings, hashmaps, two pointers, sliding window
Should know        Trees, graphs, BFS/DFS, recursion
Good to have       Dynamic programming, heaps, tries
```

Don't start with hard problems. Start with easy, get the patterns, move to medium. Most interviews are medium difficulty.

## System design (for senior roles — less relevant for new grads)

Most new grad interviews don't include system design. If they do:
- **System Design Interview** by Alex Xu (book) — the standard reference
- **Grokking System Design** (Educative) — structured course

## Data Science specific

In addition to coding, expect:
- Statistics and probability questions (distributions, A/B testing, p-values)
- ML concepts (bias/variance tradeoff, overfitting, model evaluation metrics)
- Case studies — "how would you build a recommendation system?"
- SQL (always)

Resources:
- **StatQuest** (YouTube) — statistics and ML concepts explained clearly
- **Ace the Data Science Interview** (book) — covers SQL, stats, ML, and cases

---

## Data Engineering specific

Expect:
- SQL — window functions, CTEs, optimization
- Python — data processing, pandas, PySpark basics
- System design for data pipelines
- Specific tools (dbt, Airflow, Spark, cloud platforms)

Resources:
- **DataLemur** — SQL interview questions from real companies
- **Fundamentals of Data Engineering** (book) by Joe Reis — good conceptual foundation

---

## During the interview — how to behave

1. **Clarify before coding** — ask about input constraints, edge cases, expected output
2. **Talk through your approach** before writing a line of code
3. **Start with brute force** — state it, then optimize
4. **Test your code** with an example before saying you're done
5. **If you're stuck** — say what you know, not what you don't. "I know this involves a hashmap, let me think through the structure..."

Silence is worse than thinking out loud wrong. Keep talking.
